# Design Document: fix-auth-schema-generation

## Overview

The four Better Auth schema files in `packages/db/src/schema/` were written by hand and have accumulated drift from what Better Auth 1.6.11 actually expects. The most critical mismatch is the `id` column type: the current files use `uuid` with `defaultRandom()`, but Better Auth generates 32-character alphanumeric text IDs — a fact already acknowledged by migration `0003_text_ids.sql`, which altered the live database columns to `text`. The Drizzle schema files were never updated to match.

This design covers:
1. Deriving the canonical Better Auth field list via `getSchema` from `better-auth/db`.
2. Rewriting all four schema files to match that canonical list, using `text` IDs throughout.
3. Preserving the four app-specific columns on `users`.
4. Verifying that `relations.ts` and `schema/index.ts` remain correct.
5. Generating and applying a Drizzle migration.
6. Confirming the TypeScript build is clean.

## Architecture

The change is entirely within `packages/db`. No runtime logic changes — only schema definitions and a database migration.

```
packages/db/
├── src/
│   └── schema/
│       ├── users.ts          ← rewrite
│       ├── sessions.ts       ← rewrite
│       ├── accounts.ts       ← rewrite
│       ├── verifications.ts  ← rewrite
│       ├── relations.ts      ← verify / update if needed
│       └── index.ts          ← verify exports (no change expected)
└── migrations/
    └── 0004_<name>.sql       ← new migration generated by drizzle-kit
```

### Key Differences from Current Manual Files

| Table | Current | Canonical |
|---|---|---|
| All `id` columns | `uuid().primaryKey().defaultRandom()` | `text('id').primaryKey()` |
| All `userId` FK columns | `uuid(...)` | `text(...)` |
| `sessions.token` | `varchar(512)` | `text` |
| `sessions.ipAddress` | `varchar(45)` | `text` |
| `sessions.userAgent` | `varchar(512)` | `text` |
| `accounts.accountId` | `varchar(255)` | `text` |
| `accounts.providerId` | `varchar(255)` | `text` |
| `accounts.scope` | `varchar(512)` | `text` |
| `verifications.identifier` | `varchar(255)` | `text` |
| `verifications.createdAt` | not null | nullable |
| `verifications.updatedAt` | not null | nullable |

> **Note on `varchar` vs `text`**: Better Auth's internal schema uses generic `string` type. The Drizzle field converter maps this to `text` (unbounded). The current files used `varchar` with explicit lengths. Switching to `text` is correct and matches what Better Auth's own CLI would generate. The live database already stores these as `text` after migration `0003`.

### `relations.ts` Impact

The `relations.ts` file references `users.ownerAccountId` and `users.id`. Both columns are preserved in the new schema (same names, same types). No changes to `relations.ts` are required.

### `schema/index.ts` Impact

The barrel file re-exports all four tables by name. Since the exported variable names (`users`, `sessions`, `accounts`, `verifications`) are unchanged, no edits to `index.ts` are required.

## Data Models

### New `users.ts`

```typescript
import { boolean, pgTable, text, timestamp, uuid, varchar } from 'drizzle-orm/pg-core'

export const users = pgTable('users', {
  id: text('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull().unique(),
  emailVerified: boolean('email_verified').notNull().default(false),
  image: text('image'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).notNull(),
  // App-specific fields
  role: varchar('role', { length: 20 }).notNull().default('owner'),
  ownerAccountId: uuid('owner_account_id'),
  phone: varchar('phone', { length: 20 }),
  languagePreference: varchar('language_preference', { length: 5 }).default('bn'),
})
```

### New `sessions.ts`

```typescript
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core'
import { users } from './users'

export const sessions = pgTable('sessions', {
  id: text('id').primaryKey(),
  expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(),
  token: text('token').notNull().unique(),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).notNull(),
  ipAddress: text('ip_address'),
  userAgent: text('user_agent'),
  userId: text('user_id')
    .notNull()
    .references(() => users.id, { onDelete: 'cascade' }),
})
```

### New `accounts.ts`

```typescript
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core'
import { users } from './users'

export const accounts = pgTable('accounts', {
  id: text('id').primaryKey(),
  accountId: text('account_id').notNull(),
  providerId: text('provider_id').notNull(),
  userId: text('user_id')
    .notNull()
    .references(() => users.id, { onDelete: 'cascade' }),
  accessToken: text('access_token'),
  refreshToken: text('refresh_token'),
  idToken: text('id_token'),
  accessTokenExpiresAt: timestamp('access_token_expires_at', { withTimezone: true }),
  refreshTokenExpiresAt: timestamp('refresh_token_expires_at', { withTimezone: true }),
  scope: text('scope'),
  password: text('password'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).notNull(),
})
```

### New `verifications.ts`

```typescript
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core'

export const verifications = pgTable('verifications', {
  id: text('id').primaryKey(),
  identifier: text('identifier').notNull(),
  value: text('value').notNull(),
  expiresAt: timestamp('expires_at', { withTimezone: true }).notNull(),
  createdAt: timestamp('created_at', { withTimezone: true }),
  updatedAt: timestamp('updated_at', { withTimezone: true }),
})
```

## Correctness Properties

This feature involves schema definition changes (IaC-like declarative configuration) and database migrations. The acceptance criteria are primarily structural checks (does column X have type Y?) and CLI command outcomes (does `db:generate` succeed?). These are not suitable for property-based testing:

- Schema structure checks are deterministic and don't vary meaningfully with input — 100 iterations would find no more bugs than 1.
- Migration commands test external tooling (drizzle-kit) and a live database, not our own logic.
- There are no pure functions with a wide input space to test.

PBT does not apply to this feature. Testing strategy uses example-based unit tests for schema structure and smoke/integration tests for the migration commands.

## Error Handling

**Scenario: `db:generate` produces no migration**
- Cause: Drizzle is not detecting differences, possibly because the schema index is not being read correctly or the column types are already aligned.
- Resolution: Verify `drizzle.config.ts` points to `./src/schema/index.ts`. Confirm the new schema files are exported from the index. Check that `drizzle-kit` version (0.31.10) is compatible with the schema syntax used.

**Scenario: `db:migrate` fails with type conflict**
- Cause: Changing `varchar(N)` to `text` or `uuid` to `text` on a column that has a constraint or index may require explicit casting.
- Resolution: Inspect the generated SQL. If needed, add `USING column::text` casts manually to the migration file before re-running.

**Scenario: `check-types` fails after schema rewrite**
- Cause: A file elsewhere in the codebase references a column that was renamed or whose inferred TypeScript type changed.
- Resolution: Run `tsc --noEmit` from `packages/db`, read the error output, and fix the offending import or type assertion.

**Scenario: `verifications.createdAt` / `updatedAt` nullability change breaks existing queries**
- Cause: The current schema marks these as `not null`, but the canonical Better Auth schema marks them nullable. Any code that assumes these are always present will need a null check.
- Resolution: Search for usages of `verifications.createdAt` and `verifications.updatedAt` in the codebase and add null guards where needed.

## Testing Strategy

This feature does not use property-based testing (see Correctness Properties section). The appropriate testing approach is:

**Schema structure tests (example-based unit tests)**
- Import the new schema objects and assert that each column has the expected Drizzle column type, nullability, and default value.
- These tests run in-process with no database connection required.
- Focus on the columns that changed: `id` (text, no default), FK columns (text), and the `varchar`→`text` conversions.

**Smoke tests**
- Run `bun run db:generate` and assert exit code 0.
- Run `bun run check-types` and assert exit code 0.

**Integration tests**
- Run `bun run db:migrate` against the actual database and assert exit code 0.
- Optionally query `information_schema.columns` after migration to verify column data types match expectations.

**Manual verification**
- Start the API server and perform a sign-up and sign-in flow to confirm Better Auth can read and write all four tables correctly.
