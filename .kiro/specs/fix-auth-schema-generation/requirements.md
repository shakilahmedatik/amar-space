# Requirements Document

## Introduction

The project currently maintains four Better Auth schema files (`users.ts`, `sessions.ts`, `accounts.ts`, `verifications.ts`) that were written by hand. These files have drifted from what Better Auth 1.6.11 actually expects, causing potential runtime mismatches. The goal is to replace those four files with schema definitions that precisely match Better Auth's internal field expectations (derived programmatically via `getSchema` from `better-auth/db`), while preserving the app-specific fields on the `users` table (`role`, `ownerAccountId`, `phone`, `languagePreference`). After the schema files are updated, a new Drizzle migration must be generated and applied, and the TypeScript build must remain clean.

## Glossary

- **Better_Auth**: The authentication library (v1.6.11) used by the project, configured with the Drizzle adapter and `usePlural: true`.
- **Drizzle_Schema**: TypeScript files in `packages/db/src/schema/` that define PostgreSQL tables using `drizzle-orm/pg-core`.
- **Auth_Tables**: The four tables owned by Better Auth — `users`, `sessions`, `accounts`, `verifications`.
- **App_Fields**: Application-specific columns on the `users` table: `role`, `ownerAccountId`, `phone`, `languagePreference`.
- **getSchema**: A function exported from `better-auth/db` that returns the canonical field definitions Better Auth expects for each table.
- **Migration**: A SQL file generated by `drizzle-kit generate` and applied by `drizzle-kit migrate` that aligns the live database with the current Drizzle schema.
- **Schema_Index**: `packages/db/src/schema/index.ts` — the barrel file that re-exports all schema tables.
- **Relations_File**: `packages/db/src/schema/relations.ts` — the file that defines Drizzle ORM relations between tables.
- **Text_ID**: A `text` primary key (not UUID) used by Better Auth for its own tables, as established by migration `0003_text_ids.sql`.

## Requirements

### Requirement 1: Derive the canonical Better Auth schema

**User Story:** As a developer, I want to know exactly which fields Better Auth 1.6.11 expects in each of its tables, so that the Drizzle schema files match what the library will read and write at runtime.

#### Acceptance Criteria

1. WHEN the canonical schema is derived, THE Developer SHALL use `getSchema` from `better-auth/db` (or equivalent inspection of Better Auth internals) to obtain the authoritative field list for `user`, `session`, `account`, and `verification` tables.
2. THE canonical schema SHALL be derived with the same `betterAuth` configuration used in `apps/api/src/plugins/auth.ts` (i.e., `emailAndPassword: { enabled: true }`, `usePlural: true`).
3. WHEN the canonical field list is obtained, THE Developer SHALL document any differences between the current manual schema files and the canonical schema before making changes.

### Requirement 2: Replace the `users` schema file

**User Story:** As a developer, I want the `users.ts` schema file to exactly match Better Auth's expected user fields while retaining the app-specific columns, so that authentication works correctly and no app data is lost.

#### Acceptance Criteria

1. THE `users` table SHALL use `text` for the `id` primary key (no `defaultRandom()`), consistent with Better Auth's text ID convention established in migration `0003_text_ids.sql`.
2. THE `users` table SHALL include all fields returned by `getSchema` for the `user` model: `name` (text, not null), `email` (text, not null, unique), `emailVerified` (boolean, not null, default false), `image` (text, nullable), `createdAt` (timestamp with timezone, not null), `updatedAt` (timestamp with timezone, not null).
3. THE `users` table SHALL retain the four App_Fields: `role` (varchar 20, not null, default `'owner'`), `ownerAccountId` (uuid, nullable), `phone` (varchar 20, nullable), `languagePreference` (varchar 5, default `'bn'`).
4. WHEN the `users.ts` file is replaced, THE Schema_Index SHALL continue to export the `users` table without modification.
5. WHEN the `users.ts` file is replaced, THE Relations_File SHALL continue to compile without errors, as it references `users.ownerAccountId` and `users.id`.

### Requirement 3: Replace the `sessions` schema file

**User Story:** As a developer, I want the `sessions.ts` schema file to exactly match Better Auth's expected session fields, so that session creation and lookup work correctly.

#### Acceptance Criteria

1. THE `sessions` table SHALL use `text` for the `id` primary key (no `defaultRandom()`).
2. THE `sessions` table SHALL use `text` for the `userId` foreign key column, referencing `users.id` with `ON DELETE CASCADE`.
3. THE `sessions` table SHALL include all fields returned by `getSchema` for the `session` model: `expiresAt` (timestamp with timezone, not null), `token` (text, not null, unique), `createdAt` (timestamp with timezone, not null), `updatedAt` (timestamp with timezone, not null), `ipAddress` (text, nullable), `userAgent` (text, nullable), `userId` (text, not null, FK).
4. WHEN the `sessions.ts` file is replaced, THE Schema_Index SHALL continue to export the `sessions` table without modification.

### Requirement 4: Replace the `accounts` schema file

**User Story:** As a developer, I want the `accounts.ts` schema file to exactly match Better Auth's expected account fields, so that email/password credential storage and OAuth flows work correctly.

#### Acceptance Criteria

1. THE `accounts` table SHALL use `text` for the `id` primary key (no `defaultRandom()`).
2. THE `accounts` table SHALL use `text` for the `userId` foreign key column, referencing `users.id` with `ON DELETE CASCADE`.
3. THE `accounts` table SHALL include all fields returned by `getSchema` for the `account` model: `accountId` (text, not null), `providerId` (text, not null), `userId` (text, not null, FK), `accessToken` (text, nullable), `refreshToken` (text, nullable), `idToken` (text, nullable), `accessTokenExpiresAt` (timestamp with timezone, nullable), `refreshTokenExpiresAt` (timestamp with timezone, nullable), `scope` (text, nullable), `password` (text, nullable), `createdAt` (timestamp with timezone, not null), `updatedAt` (timestamp with timezone, not null).
4. WHEN the `accounts.ts` file is replaced, THE Schema_Index SHALL continue to export the `accounts` table without modification.

### Requirement 5: Replace the `verifications` schema file

**User Story:** As a developer, I want the `verifications.ts` schema file to exactly match Better Auth's expected verification fields, so that email verification and password reset tokens are stored and retrieved correctly.

#### Acceptance Criteria

1. THE `verifications` table SHALL use `text` for the `id` primary key (no `defaultRandom()`).
2. THE `verifications` table SHALL include all fields returned by `getSchema` for the `verification` model: `identifier` (text, not null), `value` (text, not null), `expiresAt` (timestamp with timezone, not null), `createdAt` (timestamp with timezone, nullable), `updatedAt` (timestamp with timezone, nullable).
3. WHEN the `verifications.ts` file is replaced, THE Schema_Index SHALL continue to export the `verifications` table without modification.

### Requirement 6: Update dependent files

**User Story:** As a developer, I want all files that import from the replaced schema files to continue compiling without errors, so that the rest of the codebase is not broken.

#### Acceptance Criteria

1. WHEN any column name or type changes in the Auth_Tables, THE Relations_File SHALL be updated to reflect those changes.
2. THE Schema_Index SHALL export all four replaced tables under the same exported names (`users`, `sessions`, `accounts`, `verifications`).
3. WHEN the `users` table column names are unchanged (only type changes), THE Relations_File SHALL require no structural changes.
4. IF any other file in `packages/db/src/` or `apps/api/src/` imports a column that no longer exists, THEN THE Developer SHALL update that import to use the correct column name.

### Requirement 7: Generate and apply the migration

**User Story:** As a developer, I want a new Drizzle migration to be generated and applied so that the live database schema matches the updated TypeScript schema files.

#### Acceptance Criteria

1. WHEN the four schema files have been replaced, THE Developer SHALL run `bun run db:generate` from `packages/db` to produce a new migration SQL file in `packages/db/migrations/`.
2. WHEN `db:generate` completes without errors, THE Developer SHALL run `bun run db:migrate` from `packages/db` to apply the migration to the database.
3. IF `db:generate` produces an empty migration (no changes detected), THEN THE Developer SHALL verify that the schema files are correct and that Drizzle is reading them from `./src/schema/index.ts`.
4. IF `db:migrate` fails due to a constraint violation or type conflict, THEN THE Developer SHALL inspect the generated SQL and resolve the conflict before re-running.
5. WHEN the migration is applied successfully, THE database SHALL reflect the updated column types and constraints for all four Auth_Tables.

### Requirement 8: Verify the TypeScript build

**User Story:** As a developer, I want the TypeScript build to pass after all schema changes, so that no type errors were introduced.

#### Acceptance Criteria

1. WHEN all schema files and dependent files have been updated, THE Developer SHALL run `check-types` (i.e., `tsc --noEmit`) in `packages/db`.
2. THE `check-types` command SHALL complete with zero errors.
3. IF `check-types` reports errors, THEN THE Developer SHALL fix all type errors before considering the task complete.
