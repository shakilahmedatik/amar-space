# Requirements Document

## Introduction

AmarSpace is a Bangladeshi property management platform built on a Turborepo monorepo (Bun 1.3.13, Next.js 16, Fastify 5). This document captures the requirements for seven concrete fixes and improvements: removing the unused `apps/docs` application, enabling local development port binding on the API server, making CORS configuration environment-aware, installing Tailwind v4 and shadcn/ui nova preset in the frontend, wiring DESIGN.md design tokens into the Tailwind `@theme` block, migrating all inline `style={{}}` usage to Tailwind utility classes and shadcn components, and creating the missing `.env.local` for the web application.

The goal is a codebase where the design system is expressed once (in `globals.css` `@theme`) and consumed everywhere through utility classes and shadcn primitives — no scattered inline styles, no hardcoded hex values, and no broken local development experience.

## Glossary

- **Monorepo**: The Turborepo workspace rooted at the repository root, containing `apps/*` and `packages/*`.
- **API_Server**: The Fastify 5 application in `apps/api`, responsible for all backend routes and business logic.
- **Web_App**: The Next.js 16 application in `apps/web`, serving the frontend UI.
- **CORS_Resolver**: The `@fastify/cors` origin callback function registered in `apps/api/src/app.ts`.
- **Design_Token**: A named CSS custom property defined in the `@theme` block of `apps/web/app/globals.css`, derived from `DESIGN.md`.
- **Inline_Style**: A React `style={{}}` prop containing hardcoded CSS property values directly on a JSX element.
- **Tailwind_Class**: A utility class generated by Tailwind v4 from the `@theme` block (e.g., `bg-brand-green`, `text-ink`, `rounded-full`).
- **shadcn_Primitive**: A pre-built accessible UI component from the shadcn/ui nova preset (e.g., `<Button>`, `<Badge>`, `<Card>`, `<Input>`).
- **NODE_ENV**: The `process.env.NODE_ENV` environment variable, set to `'development'` locally and `'production'` on Vercel.
- **ALLOWED_ORIGIN**: The `process.env.ALLOWED_ORIGIN` environment variable, set to the production frontend URL (e.g., `https://amarspace.com`).
- **NEXT_PUBLIC_API_URL**: The `process.env.NEXT_PUBLIC_API_URL` environment variable in `apps/web`, pointing to the API server base URL.
- **Touch_Target**: An interactive UI element (button, link, nav item) that must meet the 44px minimum height requirement from DESIGN.md.
- **Pill_Shape**: The `rounded-full` border-radius (`9999px`) applied to all buttons and badges per DESIGN.md system signature.

## Requirements

### Requirement 1: Remove apps/docs from the Monorepo

**User Story:** As a developer, I want the unused `apps/docs` application removed from the monorepo, so that the Turborepo pipeline is leaner and there are no dead workspace entries consuming build time.

#### Acceptance Criteria

1. THE Monorepo SHALL NOT contain an `apps/docs` directory after the removal is complete.
2. WHEN the monorepo root `package.json` workspace globs are evaluated, THE Monorepo SHALL NOT include any reference to `apps/docs`.
3. WHEN `turbo.json` is parsed, THE Monorepo SHALL NOT contain any pipeline task or dependency referencing the `docs` workspace package.
4. IF any file in `apps/web`, `apps/api`, or `packages/*` contains any textual reference to `apps/docs` — including import statements, comments, string literals, and documentation — THEN THE Monorepo SHALL flag it as a build error regardless of whether the reference is active code or inactive (e.g., commented-out) text.

### Requirement 2: Local Development Port Binding on the API Server

**User Story:** As a developer, I want the API server to bind to port 3001 when running locally, so that I can reach it from the frontend at `http://localhost:3001` without any manual configuration.

#### Acceptance Criteria

1. WHEN `NODE_ENV` is not `'production'`, THE API_Server SHALL call `app.listen({ port: PORT, host: '0.0.0.0' })` where `PORT` defaults to `3001` if the `PORT` environment variable is not set.
2. WHEN `NODE_ENV` is `'production'`, THE API_Server SHALL NOT call `app.listen()` and SHALL rely solely on the Vercel serverless handler export.
3. THE API_Server SHALL always export the Vercel `handler` function regardless of the value of `NODE_ENV`.
4. WHEN `app.listen()` fails to bind the port, THE API_Server SHALL log the error and exit the process with code `1`.
5. WHEN `app.listen()` succeeds, THE API_Server SHALL log the message `API listening on http://localhost:<PORT>`.

---

### Requirement 3: Environment-Aware CORS Configuration

**User Story:** As a security-conscious developer, I want CORS to be locked to specific origins per environment, so that cookie-based authentication is not exposed to arbitrary origins in production.

#### Acceptance Criteria

1. WHEN `NODE_ENV` is `'development'` and the incoming request `origin` is `http://localhost:3000` or `http://127.0.0.1:3000`, THE CORS_Resolver SHALL call the callback with `(null, true)` to allow the request.
2. WHEN `NODE_ENV` is `'development'` and the incoming request has no `origin` header (same-origin or server-to-server), THE CORS_Resolver SHALL call the callback with `(null, true)` to allow the request.
3. WHEN `NODE_ENV` is `'production'` and the incoming request `origin` matches the value of `ALLOWED_ORIGIN`, THE CORS_Resolver SHALL call the callback with `(null, true)` to allow the request.
4. WHEN `NODE_ENV` is `'production'` and the incoming request `origin` does NOT match `ALLOWED_ORIGIN`, THE CORS_Resolver SHALL call the callback with a CORS error to reject the request.
5. THE CORS_Resolver SHALL always be registered with `credentials: true` to support cookie-based authentication.
6. WHEN `NODE_ENV` is `'development'` and the incoming request `origin` is any value other than `http://localhost:3000`, `http://127.0.0.1:3000`, or absent, THE CORS_Resolver SHALL call the callback with a CORS error to reject the request.

### Requirement 4: Install Tailwind v4 and shadcn/ui Nova Preset in apps/web

**User Story:** As a frontend developer, I want Tailwind v4 and the shadcn/ui nova preset installed in `apps/web`, so that I can use utility classes and accessible UI primitives throughout the application.

#### Acceptance Criteria

1. THE Web_App SHALL have `tailwindcss` (v4 or later) and `@tailwindcss/postcss` listed as dependencies in `apps/web/package.json`.
2. THE Web_App SHALL have a `postcss.config.mjs` file at `apps/web/postcss.config.mjs` that registers `@tailwindcss/postcss` as a PostCSS plugin.
3. THE Web_App SHALL have a `components.json` file at `apps/web/components.json` configured with `"style": "nova"`, `"rsc": true`, `"tsx": true`, and `"cssVariables": true`.
4. THE Web_App SHALL have a `components/ui/` directory containing the shadcn/ui base component files generated by `bunx shadcn@latest init`.
5. WHEN `bun run build` is executed in `apps/web`, THE Web_App SHALL complete successfully with zero errors, including TypeScript errors, missing import errors, PostCSS configuration errors, and Tailwind directive errors.

### Requirement 5: Wire DESIGN.md Tokens into the Tailwind @theme Block

**User Story:** As a frontend developer, I want every DESIGN.md design token wired into the Tailwind `@theme` block in `globals.css`, so that I can use semantic utility classes like `bg-brand-green`, `text-ink`, and `rounded-full` instead of hardcoded values.

#### Acceptance Criteria

1. THE Web_App SHALL have an `@import "tailwindcss"` directive at the top of `apps/web/app/globals.css`.
2. THE Web_App SHALL have an `@theme` block in `apps/web/app/globals.css` containing all color tokens from DESIGN.md, including `--color-primary`, `--color-brand-green`, `--color-brand-blue`, `--color-canvas`, `--color-surface`, `--color-ink`, `--color-hairline`, `--color-success-bg`, `--color-success-text`, `--color-warning-bg`, `--color-warning-text`, `--color-error-bg`, `--color-error-text`, `--color-on-dark`, and `--color-footer-bg`.
3. THE Web_App SHALL have an `@theme` block in `apps/web/app/globals.css` containing all border-radius tokens from DESIGN.md with their exact values: `--radius-xs: 4px`, `--radius-sm: 6px`, `--radius-md: 8px`, `--radius-lg: 12px`, `--radius-xl: 16px`, `--radius-xxl: 20px`, `--radius-xxxl: 24px`, and `--radius-full: 9999px`; a token with an incorrect value (e.g., `--radius-xs: 5px`) SHALL NOT satisfy this criterion.
4. THE Web_App SHALL have an `@theme` block in `apps/web/app/globals.css` containing all spacing tokens from DESIGN.md: `--spacing-xxs` through `--spacing-section-lg`.
5. THE Web_App SHALL have an `@theme` block in `apps/web/app/globals.css` containing font-family tokens `--font-sans` (DM Sans, Noto Sans Bengali, Inter, system-ui, sans-serif) and `--font-mono`.
6. WHEN Tailwind v4 processes `globals.css`, THE Web_App SHALL generate utility classes for every `@theme` token (e.g., `bg-brand-green`, `text-ink`, `border-hairline`, `rounded-full`, `p-xl`).

### Requirement 6: Migrate All Inline Styles to Tailwind Classes and shadcn Components

**User Story:** As a frontend developer, I want all inline `style={{}}` props replaced with Tailwind utility classes and shadcn primitives, so that the design system is expressed consistently through the `@theme` block and there are no scattered hardcoded values.

#### Acceptance Criteria

1. WHEN all TSX files in `apps/web/components/**/*.tsx` and `apps/web/app/**/*.tsx` are scanned, THE Web_App SHALL contain zero occurrences of the `style={{` prop pattern.
2. WHEN all TSX files in `apps/web/components/**/*.tsx` and `apps/web/app/**/*.tsx` are scanned, THE Web_App SHALL contain zero raw hex color literals matching the pattern `#[0-9a-fA-F]{3,6}` outside of `globals.css`.
3. THE Web_App SHALL use the shadcn `<Button>` primitive for all button elements, with `rounded-full` applied to satisfy the DESIGN.md pill-shape system signature.
4. THE Web_App SHALL use the shadcn `<Badge>` primitive for all status indicators (active, pending, overdue, error, info, neutral), with `rounded-full` applied.
5. THE Web_App SHALL use the shadcn `<Card>` and `<CardContent>` primitives for all card surfaces (stat cards, property cards, tenant cards, payment cards).
6. THE Web_App SHALL use the shadcn `<Input>` and `<Label>` primitives for all form fields.
7. THE Web_App SHALL use the shadcn `<Skeleton>` primitive for all loading placeholder states.
8. THE Web_App SHALL use the shadcn `<AlertDialog>` primitive for all confirmation dialogs.
9. THE Web_App SHALL use the shadcn `<Table>` primitive for all data table surfaces.
10. WHEN any interactive element (button, link, nav item) is rendered, THE Web_App SHALL apply a minimum height of 44px (via `min-h-[44px]` or `min-h-11`) to satisfy the DESIGN.md touch target requirement.
11. WHEN a color value from DESIGN.md has no corresponding `@theme` token, THE Web_App SHALL use an arbitrary Tailwind value class (e.g., `bg-[#hexvalue]`) rather than a `style={{}}` prop, and the missing token SHALL be added to the `@theme` block.

---

### Requirement 7: Create apps/web/.env.local with API URL

**User Story:** As a developer, I want `apps/web/.env.local` to exist with `NEXT_PUBLIC_API_URL=http://localhost:3001`, so that the frontend can reach the local API server during development without any manual setup.

#### Acceptance Criteria

1. THE Web_App SHALL have a file at `apps/web/.env.local` containing the line `NEXT_PUBLIC_API_URL=http://localhost:3001`.
2. THE Web_App SHALL have a file at `apps/web/.env.example` (or equivalent) that documents the `NEXT_PUBLIC_API_URL` variable without committing a real value.
3. WHEN `NEXT_PUBLIC_API_URL` is not set in the environment, THE Web_App SHALL always fall back to `http://localhost:3001` as the API base URL in the `apiFetch` utility, and this fallback SHALL function correctly under all conditions.
4. IF `NEXT_PUBLIC_API_URL` is not set in the Vercel production environment, THEN THE Web_App SHALL surface a build-time or runtime warning indicating the variable is missing.
